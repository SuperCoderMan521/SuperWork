# Desktop 工作区编辑器启动器设计

## 目标

在 Desktop 文件区域提供“在编辑器中打开”菜单。用户每次点击菜单，都能看到当前 Windows 系统中已安装且可启动的常见代码编辑器，并选择其中一个打开当前会话的整个工作区目录。编辑器无需提前运行。

## 范围

支持以下编辑器：

- Visual Studio Code
- Cursor
- Windsurf
- Trae
- Zed
- Sublime Text
- Notepad++
- JetBrains 系列 IDE

首版仅实现 Windows 检测与启动。未知应用、普通文件关联程序及任意自定义可执行文件不在范围内。

## 用户交互

文件区域标题栏右侧显示“在编辑器中打开 ▾”按钮。点击按钮后在其下方展开菜单，并立即触发编辑器检测。

菜单状态：

1. 检测中：显示“正在检测编辑器…”。
2. 检测成功：只显示已验证可启动的编辑器。
3. 空列表：显示“未检测到支持的编辑器”和“重新检测”。
4. 启动中：禁用所选项并显示启动反馈，防止重复点击。
5. 启动失败：保持菜单打开，在对应项下显示错误，并允许重试。
6. 启动成功：关闭菜单。

每次展开菜单都发起后台刷新。可先显示短时缓存结果以减少等待，刷新完成后替换为最新结果。

## 架构

### 渲染层

文件面板只负责菜单状态和展示。渲染层向预加载 API 发送预定义的编辑器 ID，不接受也不传递任意可执行文件路径或命令参数。

### Electron 主进程

新增独立编辑器发现服务，负责：

- 从 Windows `App Paths` 注册表查询安装信息。
- 检查用户级和系统级常见安装目录。
- 通过 `where.exe` 查询 PATH 中的编辑器命令。
- 检查 JetBrains Toolbox 与常规 JetBrains 安装位置。
- 规范化、验证并去重可执行文件路径。
- 返回稳定编辑器 ID、显示名称和图标类型。

新增受限 IPC：

- `listWorkspaceEditors()`：返回受支持且已验证的编辑器列表。
- `openWorkspaceInEditor(editorId, workspace)`：按编辑器 ID 查找主进程已验证的可执行文件，并打开工作区。

主进程不信任渲染层传入的工作区路径。启动前必须解析绝对路径、确认其为现有目录，并确保它与当前已知会话工作区一致。

### 启动行为

使用无 shell 的子进程启动方式，将工作区目录作为独立参数传入，避免命令拼接。编辑器未运行时启动新进程；编辑器已经运行时是否复用窗口由编辑器自身处理。

## 检测优先级

每个编辑器按以下顺序寻找第一个有效可执行文件：

1. Windows `App Paths` 注册表。
2. 编辑器已知的用户级与系统级安装路径。
3. PATH 中的官方 CLI 命令。
4. JetBrains Toolbox 和 JetBrains 产品目录（仅 JetBrains 系列）。

最终结果按产品定义顺序显示，而不是按发现顺序显示，保证菜单稳定。

## 缓存与错误处理

检测结果在主进程内短时缓存。每次展开菜单可立即返回缓存，同时启动一次刷新。检测单个来源失败不能中断整个列表；仅当所有来源都失败时返回可见错误。

启动失败返回结构化错误，不暴露敏感环境变量或完整命令行。诊断日志记录编辑器 ID、检测来源和错误码。

## 测试

- 单元测试注册表、常见路径、PATH 和 JetBrains 结果的合并、验证、去重与排序。
- 单元测试不存在的可执行文件不会展示。
- 单元测试只允许受支持的编辑器 ID，并拒绝无效工作区。
- 预加载 API 测试确保不暴露通用 IPC 或任意命令执行接口。
- UI 测试覆盖检测中、结果列表、空状态、启动中、成功关闭和失败重试。
- Windows 打包后验证至少一种已安装编辑器能打开完整工作区。

## 验收标准

- 用户每次点击按钮都会看到编辑器菜单和最新检测状态。
- 菜单只展示本机实际可启动的受支持编辑器。
- 选择编辑器后打开当前会话的整个工作区目录。
- 编辑器不需要提前运行。
- 不允许渲染层指定任意可执行文件或任意启动参数。
