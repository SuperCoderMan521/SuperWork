# SuperWork Desktop Performance Center Design

## Goal

Add a local, workspace-history performance center to SuperWork Desktop without
changing `query()`, `QueryEngine`, or the existing LoopQuery lifecycle.

The first version helps users answer four questions:

1. How many tokens, requests, sessions, and turns did this workspace consume?
2. How effective was prompt caching and what was the estimated cost?
3. Which tools and models dominate time and usage?
4. Are failures, interruptions, or missing diagnostics increasing?

## Scope

The first version is completely local and offline. It reads existing Claude Code
session JSONL files, Desktop turn usage already held in live sessions, configured
model prices, and Debug log metadata. It does not upload telemetry and does not
require Langfuse.

Included:

- workspace-history aggregation, selected by the current conversation workspace;
- date filters for 7 days, 30 days, and all time, defaulting to 30 days;
- summary cards for sessions, turns, tokens, cache hit rate, estimated cost,
  wall-clock duration, API duration when present, failures, and interruptions;
- token and duration trend series;
- token category composition;
- model usage distribution;
- tool call count and average duration when timestamp pairs are available;
- diagnostic status for Debug logs and optional Langfuse configuration;
- manual refresh and visible scan limits/warnings;
- loading, empty, partial-data, and failure states.

Excluded:

- SQLite or a permanent analytics database;
- background file watchers or continuous indexing;
- remote telemetry upload;
- launching or embedding Langfuse;
- changes to the original TUI, `query()`, or QueryEngine accounting;
- automatic performance recommendations generated by another model.

## Architecture

Use a dedicated `DesktopPerformanceService` in the Bun Core sidecar. Renderer
never reads session files directly.

```text
PerformanceCenterPage
  -> desktopApi.getPerformance(cwd, range, force)
  -> performance.get command
  -> DesktopCommandDispatcher
  -> DesktopPerformanceService.snapshot()
  -> Claude session JSONL + model pricing + debug metadata
  -> performance.snapshot event
  -> PerformanceCenterPage
```

The service is read-only. It receives injected filesystem/session-location
dependencies where practical so aggregation can be tested without touching real
user data. A short in-memory cache is keyed by normalized workspace and range.
`force: true` bypasses the cache for the refresh button.

## Data Sources

### Session transcripts

Reuse the same project-directory convention already used by `SessionService` and
Claude Code transcript persistence. Scan only `.jsonl` session files associated
with the selected workspace.

For every valid record, collect:

- timestamp and session identifier;
- user/assistant/system role;
- assistant usage fields;
- model name;
- tool-use blocks and tool-result blocks;
- result/error/interruption indicators when recorded.

Malformed lines are skipped and counted. A malformed line never fails the whole
snapshot.

### Model pricing

Read pricing through the existing `DesktopConfigService.modelConfig(cwd)` API.
Cost uses the same four-token-category formula as per-turn reports. Unknown or
unconfigured model prices produce partial cost data, not zero cost.

### Debug and Langfuse status

Debug status reports whether the current Claude config directory has a debug
directory, latest log path, last modified time, and file size. It does not return
log content in the performance snapshot.

Langfuse status is a boolean derived from presence of both public and secret-key
environment variables. No key values are exposed to Renderer.

## Protocol

Add a request command:

```ts
{
  type: 'performance.get'
  requestId: string
  cwd: string
  range: '7d' | '30d' | 'all'
  force?: boolean
}
```

Add a response event:

```ts
{
  type: 'performance.snapshot'
  requestId: string
  snapshot: DesktopPerformanceSnapshot
}
```

`DesktopPerformanceSnapshot` contains:

```ts
type DesktopPerformanceSnapshot = {
  cwd: string
  range: '7d' | '30d' | 'all'
  generatedAt: number
  scannedSessions: number
  scannedLines: number
  skippedLines: number
  truncated: boolean
  summary: {
    sessions: number
    turns: number
    messages: number
    apiCalls: number
    tokens: DesktopTokenUsage
    cacheHitRate?: number
    estimatedCostUsd?: number
    pricedTokenShare: number
    wallClockMs: number
    apiDurationMs?: number
    failedTurns: number
    interruptedTurns: number
  }
  trend: Array<{
    bucket: string
    tokens: number
    wallClockMs: number
    sessions: number
    failures: number
  }>
  models: Array<{
    model: string
    tokens: DesktopTokenUsage
    apiCalls: number
    estimatedCostUsd?: number
  }>
  tools: Array<{
    name: string
    calls: number
    completed: number
    failed: number
    averageDurationMs?: number
  }>
  diagnostics: {
    debugLogAvailable: boolean
    debugLogPath?: string
    debugLogUpdatedAt?: number
    debugLogSizeBytes?: number
    langfuseConfigured: boolean
  }
  warnings: string[]
}
```

All paths are local display paths. Tokens, secrets, prompts, assistant content,
tool inputs, and tool outputs are never included in this snapshot.

## Aggregation Rules

- A session is counted once per transcript file containing at least one valid
  record in the requested date range.
- A turn is a real user prompt. Tool-result user records and metadata-only user
  records do not increment the count.
- Token categories are summed from assistant usage records. Missing fields are
  zero; negative and non-finite values are ignored.
- Cache hit rate is
  `cacheRead / (input + cacheCreation + cacheRead)`. It is absent when the
  denominator is zero.
- Tool duration uses matching `tool_use.id` and `tool_result.tool_use_id`
  timestamps. Unmatched calls still count, but have no duration.
- Wall-clock duration is the sum of each session's last timestamp minus first
  timestamp inside the selected range.
- Trend buckets are daily for 7d and 30d, monthly for all time.
- Costs are grouped by model. Models without configured prices are excluded from
  `estimatedCostUsd`; `pricedTokenShare` makes partial estimates explicit.

## Scan Limits and Caching

To protect the Bun sidecar from large history directories:

- maximum transcript files per scan: 500, newest first;
- maximum lines per file: 20,000, newest lines retained;
- maximum aggregate lines: 250,000;
- cache lifetime: 30 seconds per workspace/range;
- only one scan runs per cache key; concurrent requests share its promise.

When any limit is reached, `truncated` is true and a human-readable warning is
shown in the page.

## Renderer Design

Add `performance` as a third top-level App view beside `chat` and `settings`.
The left sidebar receives a `性能中心` entry between conversation and
configuration actions. Opening it keeps session state and generation running.

The page follows the approved dark SuperWork layout:

1. header: workspace, range selector, refresh button, last scan timestamp;
2. summary grid: total tokens/cache, estimated cost, total/API duration,
   failures/interruption;
3. token-and-duration trend chart using lightweight CSS/SVG, without a new chart
   dependency;
4. token composition bar;
5. tool performance table and model distribution;
6. diagnostic status and scan warnings.

The page remains usable at narrow widths: cards wrap, charts retain horizontal
scroll, and tables use fixed minimum widths. It does not affect the existing
resizable chat/file layout.

## States and Error Handling

- Loading: skeleton cards and “正在分析本地会话…” text.
- Empty: explain that no session data exists for this workspace/range.
- Partial: render available data and show warnings for limits, malformed records,
  missing prices, or missing timestamps.
- Failure: show a compact error with retry; do not replace the chat error banner.
- Refresh: preserve the previous snapshot while showing a small refreshing state.
- Workspace change: discard the old visible snapshot and request the new workspace.

## Testing

Use TDD at each boundary:

- protocol schemas accept valid performance commands/events and reject invalid
  ranges or negative metrics;
- aggregation tests cover valid records, malformed lines, tool timing, turn
  classification, cache rate, model pricing, range filters, and scan limits;
- dispatcher tests verify `performance.get` routing and failure isolation;
- preload API tests verify only the typed performance method is exposed;
- Renderer tests cover navigation, loading, empty, partial, error, range change,
  refresh, and the approved metric layout;
- Desktop typecheck, complete Desktop tests, and production build must pass.

## Privacy and Compatibility

The service reads local files only. The snapshot contains numeric aggregates and
local diagnostic metadata, never conversation content or credentials. Existing
TUI behavior and LoopQuery logic remain untouched. Protocol additions are
backward-compatible within the existing version because both Desktop processes
ship together.
